# Fail2ban

## Принцип работы

Fail2ban — это демон, который анализирует логи сервисов (например, SSH, Asterisk) на предмет подозрительной активности. Для каждого сервиса используется свой фильтр — набор регулярных выражений, которые ищут в строках лога признаки неудачных попыток (неверный пароль, подбор учётных записей и т.п.). При совпадении из строки извлекается IP-адрес (метка `<HOST>`).

Затем работает счётчик: для каждого IP фиксируется количество срабатываний за определённый промежуток времени. Если лимит превышен — IP блокируется на уровне межсетевого экрана (iptables/nftables).

В репозитории представлены фильтры:
- Asterisk: `filter.d/asterisk.conf`;
- Backend auth API: `filter.d/auth.conf`.

Для SSH используется стандартный фильтр из пакета fail2ban (обычно `/etc/fail2ban/filter.d/sshd.conf`), а лог берётся из `/var/log/auth.log`.

### Ключевые параметры настройки (задаются в `jail.local`)

- **findtime** — интервал времени в секундах, за который учитываются попытки;
- **maxretry** — количество срабатываний за `findtime`, после которого IP блокируется;
- **bantime** — длительность блокировки в секундах. Значение `-1` означает бессрочный бан.

### Откуда Fail2ban получает данные

Для каждого джейла (jail) в конфиге указывается путь к лог-файлу:
- `sshd` → `/var/log/auth.log`
- `asterisk` → `/var/log/asterisk/messages`
- `auth` → journald (`_SYSTEMD_UNIT=intercom-backend.service`)

Fail2ban читает новые записи из этих файлов в реальном времени и применяет к ним регулярные выражения из соответствующего фильтра.

## Пояснения к фильтрам auth API

Здесь разобрано, какие строки лога попадают под `failregex` в `filter.d/auth.conf`, и как это влияет на счётчик IP в соответствующем jail.

### Общее

- Это **не отдельный «системный» журнал ОС**: совпадения ищутся в тех же строках, что пишет приложение `intercom-backend` (Fastify/Pino) из `src/routes/auth.ts` в формате `[TAG] ... ip=<HOST> ...`.
- Каждая строка, совпавшая с `failregex`, даёт **+1** к счётчику данного IP в **этом** jail (до срабатывания `maxretry`).
- Строки OTP-воркера с тегами `AUTH_QUEUE_DROP_*` и `AUTH_REQ_CALL_FAILED` (`src/otp/worker.ts`) не должны входить в auth-фильтр и на счётчики не влияют.
- Тег **`AUTH_VERIFY_SUCCESS`** в фильтрах **не используется**: успешная верификация счётчик fail2ban не увеличивает.

### `filter.d/auth.conf`

Текущий auth-фильтр intentionally узкий: бан в fail2ban вызывается только явным серверным событием блокировки (`AUTH_BLOCK`), которое пишет backend при превышении внутренних порогов abuse-защиты.

Это исключает баны по успешным запросам и снижает ложные срабатывания на легитимном трафике.

## Где хранится состояние

- Текущие счётчики попыток и список заблокированных IP хранятся в памяти демона.
- Правила блокировки добавляются в отдельные цепочки iptables (например, `f2b-sshd`, `f2b-asterisk`).
- При остановке fail2ban все добавленные им правила удаляются из фаервола.
- Постоянное хранилище (база забаненных IP, позиции в логах) — SQLite-файл по пути из `dbfile` в `fail2ban.conf` (по умолчанию `/var/lib/fail2ban/fail2ban.sqlite3`). При перезапуске fail2ban подгружает данные из этой базы.

## Белый список (ignoreip)

IP-адреса и подсети, перечисленные в параметре `ignoreip`, полностью исключаются из мониторинга: они не учитываются в счётчиках и никогда не блокируются.  
При деплое через `deploy/install.sh` один список задаётся в `.env` как `FAIL2BAN_IGNOREIP` и подставляется во все jail (asterisk, sshd, auth). Если правите `jail.local` вручную на сервере, допишите нужные адреса в каждую секцию с `ignoreip`, чтобы не заблокировать себя ни в одном из сервисов.

## Настройка через `jail.local`

### Открытие и правка jail.local на сервере

На сервере конфиг fail2ban лежит по пути `/etc/fail2ban/jail.local`. Для правки нужны права root:

```bash
sudo nano /etc/fail2ban/jail.local
```

После сохранения изменений примените конфиг: `sudo fail2ban-client reload` (см. [Проверка и перезагрузка конфигурации](#проверка-и-перезагрузка-конфигурации)).

### Добавление IP в белый список

Чтобы адрес никогда не блокировался, добавьте его в параметр `ignoreip` в секциях `[sshd]` и при необходимости `[asterisk]` в `/etc/fail2ban/jail.local`. В каждой секции строка `ignoreip` одна — допишите нужный IP в конец этой строки через пробел.

Пример: добавить `192.168.1.1` в белый список SSH:

```ini
[sshd]
ignoreip = 1127.0.0.1/8 ::1 192.168.1.1
```

Если этот IP уже заблокирован, сначала разблокируйте его: `sudo fail2ban-client set sshd unbanip 192.168.1.1`, затем отредактируйте `ignoreip` и выполните `sudo fail2ban-client reload`.

- В каждой секции джейла (`[asterisk]`, `[sshd]`) параметр `ignoreip` должен встречаться **только один раз**. Наличие дубликатов приведёт к ошибке при запуске.
- Если вы используете несколько IP-адресов или подсетей, перечисляйте их в одной строке через пробел.
- Убедитесь, что в конце строки нет лишнего пробела.

### Примеры записи ignoreip

```ini
# Один адрес
ignoreip = 127.0.0.1/8 ::1 192.168.1.1

# Несколько адресов
ignoreip = 127.0.0.1/8 ::1 192.168.1.1 192.168.1.2

# Диапазон (подсеть)
ignoreip = 127.0.0.1/8 ::1 192.168.1.0/24

# Комбинация: отдельный адрес + подсеть
ignoreip = 127.0.0.1/8 ::1 10.0.0.1 192.168.1.0/24
```

## Управление сервисом

### Запуск и остановка
```bash
# Включить автозапуск и запустить
sudo systemctl enable --now fail2ban

# Полностью остановить (все текущие блокировки будут сняты)
sudo systemctl stop fail2ban
```

### Проверка и перезагрузка конфигурации
```bash
# Проверить синтаксис конфигурационных файлов
sudo fail2ban-client -t

# Применить изменения (например, после правки ignoreip)
sudo fail2ban-client reload
```

### Работа с блокировками

#### Как посмотреть список забаненных адресов

```bash
# Статус джейла SSH (количество банов и список IP)
sudo fail2ban-client status sshd

# Статус джейла Asterisk
sudo fail2ban-client status asterisk

# Статус auth API джейла
sudo fail2ban-client status auth
```

В выводе будут строки `Currently banned:` (число) и `Banned IP list:` (список адресов через пробел). Если забаненных нет, список будет пустым. «Total banned» — сколько всего раз банили за всё время; в «Banned IP list» только те IP, которые заблокированы в данный момент.

#### Посмотреть список всех забаненных адресов

```bash
for j in auth; do
  echo "=== $j ==="
  sudo fail2ban-client get "$j" banip
done
```

#### Проверка конкретного IP

Прямой команды «статус по адресу» в fail2ban нет. Можно проверить, есть ли IP в списке забаненных для джейла:

```bash
# Забанен ли 192.168.1.100 в asterisk (если в выводе есть строка с этим IP — да)
sudo fail2ban-client status asterisk | grep 192.168.1.100

# То же для SSH
sudo fail2ban-client status sshd | grep 192.168.1.100
```

Чтобы узнать, в каком джейле забанен адрес (подставьте свой IP вместо примера):

```bash
for j in sshd asterisk auth; do
  echo "== $j =="
  sudo fail2ban-client status "$j" | grep 192.168.1.100 || echo "нет"
done
```

#### Разблокировка

```bash
# Разблокировать IP в SSH
sudo fail2ban-client set sshd unbanip <IP>

# Разблокировать IP в Asterisk
sudo fail2ban-client set asterisk unbanip <IP>

# Разблокировать IP в auth API джейле
sudo fail2ban-client set auth unbanip <IP>
```

### Проверка применённых настроек

```bash
# Посмотреть, какие адреса в белом списке для SSH
sudo fail2ban-client get sshd ignoreip

# Для Asterisk
sudo fail2ban-client get asterisk ignoreip

# Для auth API
sudo fail2ban-client get auth ignoreip
```

## Сохранение базы данных fail2ban

Чтобы не терять накопленный список забаненных IP при переустановке сервера, базу можно хранить в репозитории и подменять ею пустую базу при деплое.

### Путь к файлу базы на сервере

Файл задаётся параметром `dbfile` в `/etc/fail2ban/fail2ban.conf` (по умолчанию `/var/lib/fail2ban/fail2ban.sqlite3`).

### Как сохранить базу в репозиторий

1. На **своём компьютере** скопировать файл с сервера (подставьте своего пользователя и хост):
   ```bash
   scp USER@YOUR_SERVER_IP:/var/lib/fail2ban/fail2ban.sqlite3 ./configs/fail2ban/fail2ban.sqlite3
   ```
2. Закоммитить и запушить `configs/fail2ban/fail2ban.sqlite3` в репозиторий.
3. Периодически повторять шаги 1–2, чтобы актуализировать базу в репо.

При деплое скрипт установки подменит пустую базу на сервере этим файлом (если он есть в репо), и fail2ban при старте подгрузит накопленные баны.
